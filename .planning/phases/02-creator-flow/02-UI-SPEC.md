## PhotoUploader Component Contract

**File:** `client/src/components/create/PhotoUploader.jsx`

### Visual Specification

- **Drop zone:** Dashed border (`2px dashed var(--border)`), cream background, rounded corners (`12px`), min-height 80px, padding 12px 18px
- **Drop zone hover/active state:** Border color changes to `var(--mint)`, background stays cream
- **Thumbnail grid:** Flexbox, wrap, gap 10px (desktop) / 8px (mobile)
- **Thumbnail size:** 80px x 80px (desktop), 64px x 64px (mobile, breakpoint 480px)
- **Thumbnail image:** `object-fit: cover`, `border-radius: 8px`
- **Remove button:** 20px x 20px circle, `var(--coral)` background, white "x" text (not an emoji), positioned top-right of thumbnail with 4px inset
- **Remove button touch target:** The visual button is 20px but the clickable area MUST be at least 44x44px (use padding or `::before` pseudo-element). Per ui-ux-pro-max skill: `touch-target-size` minimum 44x44px
- **File name display:** NOT shown per D-05 (thumbnails only with remove buttons). The original code showed file names; the new spec shows thumbnails only.
- **Counter text:** "{count}/5" shown in the label when photos > 0
- **Helper text:** "JPG, PNG, WebP, or GIF -- max 5MB each" below the drop zone in caption size

### Interaction Specification

- Click drop zone or drag files onto it to add photos
- Native `<input type="file" accept="image/jpeg,image/png,image/webp,image/gif" multiple>` behind the scenes
- Client-side validation: check count (max 5 total), file size (max 5MB), file type (MIME accept attribute as UX convenience only; real validation is server-side per photo-upload-security skill)
- On validation failure: set inline error message below the drop zone, do not add the file
- On remove: call `URL.revokeObjectURL()` on the thumbnail URL to prevent memory leak, remove from state
- `react-dropzone` is NOT installed. Use native file input with a styled label/button and `onDragOver`/`onDrop` handlers on a wrapper div.

### Accessibility

- Drop zone wrapper has `role="button"` and `tabIndex={0}` if not using a native `<label>`
- Remove buttons have `aria-label="Remove photo {n}"`
- File input has `aria-label="Upload photos"`

- **Body text:** 16.8px (1.05rem), opacity 0.8. Recipient name in `<strong>`.

- **"Create another wish" link:** Text link, no background, `var(--mint)` color, Nunito weight 700, 14.4px. On hover: underline.

- Success heading is an `<h1>` (page heading semantics)
- Copy button `aria-label`: "Copy wish link to clipboard"

## Form Validation Contract

### Validation Approach (D-11, D-12, D-13)

- **Inline validation**: Errors displayed directly below each field, not in a top-level error banner
- **Validation timing**: On submit attempt (not on blur, not on change). If the form has been submitted once and failed, subsequent changes to the errored field clear that field's error immediately.
- **Error state**: Red text (`#c0392b`), weight 700, 14px, Nunito font. Positioned below the input/select with 4px top margin.
- **Error indicator on field**: The errored input gets a red outline (`2px solid #c0392b`) instead of the default mint focus ring.

### Required vs Optional Fields

| Field | Required | Validation |
|-------|----------|-----------|
| Recipient name | YES (D-12) | Non-empty after trim |
| Birth month | YES (D-12) | Integer 1-12 |
| Birth day | YES (D-12) | Integer 1-31 |
| Message | YES (D-12) | Non-empty after trim |
| Sender name | NO (D-13) | No validation. If empty, wish is anonymous. |
| Relationship | NO (D-13) | No validation. Defaults to "friend". |
| Photos | NO (D-13) | 0-5 files, 5MB each, accepted image MIME types |

### Inline Error Display

Each field manages its own error string. A field-level `errors` state object tracks `{ fieldName: errorMessage | null }`. On submit, all required fields are validated and errors are set. On field change after first submit, that field's error is cleared.

## State Management Contract

### CreatePage State Shape

```
form: {
  senderName: string,      // optional
  recipientName: string,   // required
  relationship: string,    // optional, default "friend"
  month: number,           // required, default 6
  day: number,             // required, default 15
  message: string,         // required
  theme: string,           // required, default "sunrise"
}
photos: File[]             // 0-5 File objects
photoUrls: string[]        // Object URLs for thumbnails (revoked on removal)
errors: {                  // per-field validation errors
  recipientName: string | null,
  month: string | null,
  day: string | null,
  message: string | null,
  photos: string | null,   // upload-related errors
  global: string | null,   // API-level errors
}
loading: boolean           // true during upload + wish creation
uploadProgress: string | null  // progress message
createdWish: object | null     // wish response from API
copied: boolean            // clipboard copy feedback
submitted: boolean         // true after first submit attempt (enables per-field error clearing)
```

### Data Flow

1. User fills form, optionally adds photos, selects theme
2. On submit: validate all required fields. If errors, set `errors` and `submitted: true`, abort.
3. If valid: set `loading: true`, upload photos first via `POST /api/upload`, then create wish via `POST /api/wish`
4. On success: set `createdWish` to response, set `loading: false`
5. On error: set `errors.global` to API error message, set `loading: false`
6. Success state renders `SuccessState` component, replacing the form

## Security Contract

Per photo-upload-security skill (mandatory):

| Security Layer | Where | Implementation |
|---------------|-------|----------------|
| Client validation | PhotoUploader component | MIME accept attribute, file count check, file size check. UX convenience only. |
| Server validation | `server/middleware/upload.js` | Multer limits (5MB, 5 files), UUID filenames via `crypto.randomUUID()` |
| Magic-byte validation | `server/middleware/upload.js` | `file-type` package validates actual file content. Reject non-image. |
| Controlled serve route | `server/routes/photos.js` | UUID regex validation, path traversal prevention, resolved path check |
| Photo count enforcement | `server/services/wishService.js` | Reject `photoUrls.length > 5` at wish creation |

**No changes to server middleware or routes are expected in this phase.** The existing `server/middleware/upload.js` and `server/routes/photos.js` already implement all security requirements. Phase 2 focuses on the client-side components.
